# ADR 0002:混合检索融合策略选用 RRF

- **状态**:已接受
- **日期**:2026-06-27
- **相关文档**:[modules/memory/retrieval.md](../modules/memory/retrieval.md)

## 背景

混合检索需要把向量召回和 BM25 召回两路结果合并成单一排序。两路的打分量纲不同(cosine 相似度 vs BM25 分数),需要一个融合策略。

## 候选方案

1. **RRF(Reciprocal Rank Fusion)**:只依赖各路排名,对每个文档累加 `w/(k+rank)`。
2. **加权分数融合**:对各路原始分数归一化后加权求和。
3. **LR 校准融合**:把 cosine 与 BM25 校准到同一概率尺度再比较(EverOS 的做法)。

## 结论

第一阶段选用 **RRF**。加权融合保留为接口参数(`weights`)作备选;LR 校准列为后续扩展位。

## 理由

- **跨路天然可比**:RRF 只看排名,不需把量纲不同的 cosine 和 BM25 强行归一化对齐。
- **鲁棒、易调**:只有一个常数 `k`,不像加权融合需调权重 + 归一化方式。
- **与库默认一致**:RRF 是 LanceDB hybrid search 的默认融合器(`RRFReranker()`),减少惊讶。
- **透明可测**:实现简单(纯函数),放在仓库内可读可改,符合"算法在仓库内"的取舍。
- LR 校准更精细,但其参考实现(EverOS)在私有二进制包中、内部实现不可得;且 MVP 阶段 RRF 已足够,避免过早优化。

## 影响

- 融合逻辑在 `modules/memory/retrieval/fusion.py`,纯计算、同步函数,易单测(覆盖率门槛 ≥ 80%)。
- 倾向应用层自己做双路召回 + RRF,而非绑定 LanceDB 内置 hybrid API,保持融合逻辑可换。
- 未来若真实数据证明 RRF 不够,可启用 `weights` 加权,或引入 LR 校准融合(新增 ADR)。
