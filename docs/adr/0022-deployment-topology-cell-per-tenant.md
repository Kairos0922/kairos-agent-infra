# ADR 0022:生产部署拓扑——每租户一个 Runtime cell

- **状态**:已接受
- **日期**:2026-07-07
- **相关文档**:[ADR 0021](./0021-rust-runtime-ts-ui-architecture.md)、[ADR 0001](./0001-vector-store-lancedb.md)、[ADR 0013](./0013-lancedb-tenant-physical-tables.md)、[ADR 0019](./0019-language-migration-python-to-rust-ts.md)、[project/architecture.md](../project/architecture.md)
- **上位关系**:细化 ADR 0021「Runtime 即服务,永远一份」的**部署形态**——把"永远一份"明确为"一份 Runtime *实现*,按租户(机构)部署为相互隔离的 cell"。ADR 0021 的进程模型(单一 Rust 进程 + MCP/sandbox 子进程)、仓库结构、依赖强制结论不变,仅生产部署拓扑由本 ADR 明确(见 0021 追记)。

## 背景

目标形态已明确:服务垂直行业机构(教育:学校),**租户 = 机构、量级数十~数百,每租户多个用户(教师),总用户 1000+**。三条既有决策各自以"第一阶段单机"为前提独立论证,从未合到"1000+ 机构 SaaS"终局一起复核:

- ADR 0021:「Runtime 永远一份 + 所有入口皆客户端」。
- ADR 0001:嵌入式 LanceDB(进程内 + 本地文件)。
- ADR 0013:租户物理分表 + `drop_table` 合规删除。

若把"永远一份"读作"全局唯一一个进程",三者在目标规模相互矛盾:进程内嵌入式 LanceDB 绑定本地文件 → 起不了第二个实例读同一份数据 → **无法水平扩展、无法滚动发布(每次发版全租户停机)、单盘单进程是全租户级单点故障、`panic=abort` 一次崩溃打穿所有机构**。嵌入式存储与"全局单进程"在这个规模上互斥,必须在写 server/存储代码前把拓扑定死。

## 候选方案

| | 候选 A:每租户一个 cell(**选定**) | 候选 B:全局单一 Runtime 进程 | 候选 C:全局单进程 + client-server 向量库 |
|---|---|---|---|
| 进程模型 | 一份 Runtime 实现,每机构一个隔离实例(cell)+ 前置路由 | 全机构共用一个进程 | 全机构共用一个进程 |
| 存储 | 每 cell 自带嵌入式 LanceDB + 本地卷(ADR 0001 成立) | 嵌入式无法多实例共享,**矛盾** | 需独立向量库服务(**推翻 ADR 0001**) |
| 水平扩展 | 加 cell 即扩容,天然按机构分片 | 不能(嵌入式绑本地文件) | 能,但要额外运维向量库集群 |
| 故障爆炸半径 | 单机构 | 全租户 | 全租户(计算面) |
| 滚动发布 | 按 cell 灰度,单机构短暂不可用 | 全租户停机 | 全租户停机(计算面) |
| 合规删除 | 销 cell 卷 / `drop_table`,单机构完成(ADR 0013 成立) | `drop_table` 但共享进程 | 条件删除 + 彻底性校验 |
| 运维复杂度 | 需路由 / onboarding / 编排层(新增) | 最简,但不可扩展 | 需运维向量库集群 + 编排 |

## 结论

生产拓扑采用**每租户(机构)一个 Runtime cell**:

- **「Runtime 永远一份」= 一份 Runtime *实现*,不是一个全局进程**。同一份 Rust 二进制,按机构部署为相互隔离的 cell(进程 / 容器),各自持有该机构的嵌入式 LanceDB 本地卷与 session / trace 存储。
- **前置一层路由 + onboarding / 编排**:请求按 `tenant_id` 路由到对应 cell;新机构 = 起一个 cell + 建卷,注销 = 停 cell + 销卷。"Cloud" 客户端面对的是这层路由,不是某一个进程。
- **隔离、爆炸半径、备份、扩展、发布全部变成"每机构有界"**:一个机构的故障 / 发布 / 数据删除不波及其它机构;扩容 = 加 cell;备份 = 每 cell 卷快照。
- **本 ADR 使 ADR 0001(嵌入式 LanceDB)、ADR 0013(物理分表 + `drop_table` 合规)在目标规模成立**——它们原本只在"单机"前提成立,cell-per-tenant 把该前提落到"每机构一个单机 cell"。

## 理由

- **让已选存储决策从隐患变优势**:嵌入式 LanceDB + 本地文件 + `drop_table` 合规删除,在 cell-per-tenant 下不再是"不可扩展的单点",而是"每机构自包含、易备份、易销毁"的正解。这比推翻 ADR 0001 改用 client-server 向量库(候选 C)代价更小,且契合机构级租户量(数十~数百 cell,非百万)。
- **爆炸半径与租户边界对齐**:机构是信任边界(ADR 0010/0013),cell 是物理边界,两者重合——最危险的跨租户泄漏 / 连带故障用进程边界堵死。
- **代价可控**:代价是新增路由 / onboarding / 编排层;在数十~数百机构量级,这是标准的 cell/pod-per-tenant 运维,不引入向量库集群那类重型分布式存储运维。

## 反方观点(诚实记录)

- **"cell 数随机构数线性增长,运维负担"**:回应——目标是机构租户(数十~数百),非百万级 C 端;该量级 cell 编排是成熟实践。真到数千机构再评估 cell 内多租户合并或分层路由(另开 ADR)。
- **"全局单进程更省资源"**:回应——省的是资源,赔的是可扩展 / 可用性 / 爆炸半径;对有 SLA 与合规责任的机构 SaaS,后者优先级更高。资源可由 cell 密度(单机多 cell)平衡。

## 影响

- **architecture.md**:§1「Runtime 即服务,永远一份」改述为"一份实现、每租户一个 cell",图注同步;补路由 / 编排层位置说明。
- **ADR 0021**:加追记,指向本 ADR(进程模型 / 仓库 / 依赖结论不变,仅部署拓扑明确)。
- **ADR 0001 / 0013**:加追记,说明其"单机嵌入式"前提由 cell-per-tenant 承载,在目标规模成立。
- **server 层(未落地)**:未来需实现 `tenant → cell` 路由、cell 生命周期编排(onboarding / offboarding)、per-cell 备份;这些在 server / 运维层,不改 L0–L3 契约。
- **本地端形态**:本地单用户即"一个 cell"退化情形,天然复用同一实现(与 ADR 0019 追记的构建档一致)。
