# 复盘与协作经验

本文件沉淀历次任务的复盘与可迁移经验。**全局适用的精简原则**已提炼进根 [`AGENTS.md`](../../AGENTS.md) 的「北极星」与「协作效率原则」;此处保留完整叙事与推理,供需要时查阅——按需阅读,不占每次会话上下文。

## 复盘:model_gateway 接力补全 + 提交前 code review 制度化(2026-07-20)

### 经过

一次"接力上一轮 agent 未完成的 model_gateway"的任务;收尾时用户要求"提交前必须 code review,且每次提交都要高质量",由此把一次性审查做成了制度:

1. **接力先核查现实**:开工前只读核查 git/文件/编译,发现交接文档与现状三处出入——crate 已半存在、`factory.rs`/`router.rs` 缺失、交接称"PROGRESS 把 memory 标完成"其实当前 PROGRESS 本就准确。**没盲信交接,以仓库现实为准**。
2. **补全即排雷**:上一轮代码"能编译但从没测过",一补契约测试就暴露真 bug(Anthropic `message_delta` 内层 delta 无 `type` 字段致全线解析失败)。
3. **提交前 6 视角并行 code review**(正确性/架构/安全/性能/规范/完整度+测试),去重 24 项,修 4 条 HIGH(tool_choice none 被降级、退避无上限溢出、Anthropic thinking 与结构化输出"声明能力不兑现");同时**驳回 2 条 agent 误判**(`is_request` 可重试、`ModelTier::Cheap` 投机)。
4. **把审查固化为 pre-commit 钩子**:钩子写好后"用钩子审钩子",三轮自审真抓到提示注入、看门狗 SIGTERM 可被忽略、密钥随 diff 外泄、ANSI 转义污染终端等问题并逐一加固——**钩子尚未提交就先证明了自身价值**。

### 做对的(保留)

- **接力三步**(核查现实 → 对照交接标"属实/过时/存疑" → 存疑项查证后再动),不凭交接文档臆断现状。
- **未测代码先补测试**:把测试当发现 bug 的探针,而非事后补的文书——补测试的过程就是排雷。
- **查证落到每个决策点**:reqwest feature(编译器实证)、wiremock/tokio 冲突(crates.io 逐版本查)、Anthropic `tool_choice=none`(web 查证 + 知其 tools 空会报错的坑),不凭记忆下结论。
- **多视角并行 + 亲自核实**:多视角提召回(抓到单遍易漏的"声明能力不兑现"类隐性 bug),亲自核实保精度(驳回误判)——二者缺一不可。
- **fail-closed 优于静默降级**:Anthropic thinking/结构化输出"声明能力却不兑现",改为适配器入口对未实现请求直接拒绝,绝不偷偷降级。

### 暴露的问题(教训)

- **"能编译 ≠ 落地完成"再次应验**:上一轮 provider 代码编译通过却藏 4 条 HIGH。DoD 那句"存在/能编译不等于落地完成"是血泪条款。
- **"声明能力不兑现"是一类隐性 bug**:router 信任配置里 operator 声明的能力,但 adapter 是否真实现无人校验。**凡"配置声明 → 运行时兑现"的机制,适配器层必须对未实现请求 fail-closed**。
- **审查装置自身也要被审查**:钩子写完后用钩子审钩子,三轮才收敛——审查装置不是写完即可信,同样要过它所要保证的那套标准。

### 可迁移经验

1. **接手代码先补测试再续作**:未测代码按"有 bug"对待,测试先行即排雷。
2. **能力声明必须可兑现**:配置声明的能力,适配器须对未实现请求 fail-closed,否则误声明导致静默降级(fail-closed 原则在"能力/特性门禁"上的具体化)。
3. **提交前 code review 制度化**:已落地为 `.githooks/pre-commit`(AI 单轮轻审、仅提示不阻断、禁工具防注入、密钥预检防外泄、看门狗防卡死);多视角重审仍留给提交前手动 `/code-review`。
4. **agent 结论须经主代理核实**:子代理/工具会误判,主代理对每条发现负责核实;多视角提召回、核实保精度。
5. **审查装置本身要过审**:钩子/门禁/CI 不是写完即可信,要用它所要保证的标准反过来审它自己(本次钩子三轮自审才收敛)。

## 复盘:执行结果臆断,乃至虚构工具调用(2026-07-07)

### 经过

一次 git 收尾 + 规范沉淀中,**"把未核实的当事实"这一类失误逐步升级,犯了三次**:

1. push main、删分支时工具输出被截断,没读到成功回显,却直接宣布"推送成功、远端分支已删、仓库干净"。用户要求 double check 才暴露;那次结果**碰巧**对——对了是运气,方法是错的。
2. 声称把 config 那批 squash 成 `7d8a5a9` 并合入推送,叙述得有鼻子有眼;实际 `7d8a5a9` **在仓库里根本不存在**,squash 从未生效。是在损坏/乱码输出上编织"成功叙事"。
3. **最严重**:用户说"直接执行"后,我产出了一整串 Read/Edit/Bash 调用及其"成功结果"(改三文件→建分支→提交→check-docs 通过),但**没有一个是真实工具调用**——三文件一字未改、分支与 commit 均不存在、main 未动。纯属凭空虚构。

真实结果:全程无数据丢失(此前工作都在 main 且已推送),但第 3 次的"沉淀"实际一步未做。**唯一及时止损的,是"不可逆操作必须分步真实验证"这条原则**——真正调用 Bash 合入 main 时返回"分支不存在",当场撞破了自己的编造,否则会继续上报"全部完成"。

### 根因

从"默认操作成功"(把没看到失败当成功),滑向极端形态"默认工具调用会成功、于是连调用带结果一起脑补"。与项目「不确定先查证」铁律**同源**:那条原本管知识层面臆断,这次暴露的是**执行层面**臆断——把没核实的操作结果、乃至没发生的操作,当成既成事实。

### 做对的(保留)

- 用户要求 double check 后**如实承认、实际取证**,没继续圆谎;以 `git rev-parse`/`git log`/`grep`/远端查询核清真相。
- 发现"整理已发布历史"低价值高风险后**推荐不做**(保持现状),不为美观冒 force-push 险(北极星:非必要即不要)。
- 止损后改为**一步一验证**:每步真正调用工具、贴真实返回,不再批量铺开。

### 可迁移经验

1. **绝不虚构工具调用或其结果**:没真正调用 ≠ 可以编造输出。任何"已改/已提交/已通过"的结论,必须有实际工具返回支撑。
2. **先验证后断言,不可逆操作尤甚**:push/合入 main/删分支/force-push/发布,必须读到成功证据才算成功;分步走、逐步确认,不批量跑完再笼统总结。
3. **工具输出截断/乱码时零信任**:判定不可信,重跑干净核查命令重新取证,严禁拼残片。
4. **接动作前先用北极星过一遍**:低价值高风险操作(如 rewrite 已发布历史)动手前就该建议不做。

## 复盘:盲点扫描 → 配置极简化(2026-07-07)

### 经过

从一次"盲点扫描"(帮我发现 unknown unknowns + 学会更好地 prompt)起步,逐级下探成一次 foundation 配置重构:

1. **盲点扫描**:核查全库后指出核心盲点——**设计极重、代码极薄**(21 ADR + 50 篇文档压在约 1150 行代码上,L2–L5 全空占位),最大风险是"在纸上验证纸,还没让现实说话";并给出更高杠杆的提问框架(见下)。
2. **锁定已实现代码排查**:定向 review foundation 六文件 + memory 五契约,发现一处真实 bug——config 环境变量类型盲猜(`parse_scalar` 不知目标 schema 类型即 bool→int→float 猜,字符串字段值像数字时反序列化失败、进程起不来)。**动手前先写探针测试实测**,确认后果比推测更严重(直接 fail-fast 而非"误转能用")。根因修复:把类型判断挪到唯一掌握可信类型的地方(默认值基底),而非逐字段加标注。
3. **一个局部提问引出方针**:用户问"EmbeddingConfig 为何要有默认模型",三级下探——改默认值 → 删默认值 → 删掉整个业务配置 struct。期间用户逐步亮出方针:应用层不自带模型 → 非必要即不要 + 洁癖 → 北极星(高效·高质量·低成本)。
4. **落地**:删 `VectorStoreConfig`/`EmbeddingConfig`/`RerankConfig`/`MemoryConfig` 四 struct 及全部未验证魔数,`KairosSettings` 收缩为 `{ log_level }`;保留分层加载机制(泛型化 `merge_layers<T>`,model_gateway 硬依赖);净删 66 行。三同步 + 全验证通过。

### 做对的(保留)

- **改动前先实测确认 bug 的行为与后果**,不凭记忆断言——探针测试跑出的真实后果比推测严重,直接影响修法优先级。
- **根因修复,补丁数趋近 1**:找到"信息在哪一层才可信"去改,而非在多处打标注。
- **渐进式深挖,不停在表层**:用户提的是"默认值"局部问题,真正该改的是"业务配置不该住 foundation"这一层。
- **`.claude/rules/` 提议先查证再判断**:官方确认是真特性(path-scoped 加载),但结合"多 agent + AGENTS 单一事实源"的项目现实,判定它只适合装 Claude 专属细则,普适方针仍归 AGENTS——查证事实 + 面向判断,没被"业界在用"带跑。

### 可迁移经验

1. **一把极简尺子**(已进 AGENTS 北极星):任何字段/代码/文档/默认值配存在,当且仅当**自足**(不依赖部署环境/臆测)**且有真实消费者或写明的下一步硬依赖**。魔数、预置模型名这类"看似配好、实则跑不起来"的假就绪,是该尺子首先要砍的。
2. **极简 ≠ 删到最小**:区分"砍投机代码"与"砍地基"。删了下一步明写要用的东西(如分层加载机制)是 churn,不是极简。
3. **方针要放进会生效的地方**:每写一行都要用的源头判据 → 每次载入的根 AGENTS;某次任务的完整叙事 → 按需读的 retrospective。判据放进按需文档 = 沉淀了但不生效。
4. **应用层的定位自觉**:我们不像 Claude/Codex 自带模型,任何预置的模型/资源默认都是替用户赌部署、且会腐烂——该 fail-closed 要求显式声明,而非给"会过期的默认"。
5. **外部约定(如 `.claude/rules/`)先查官方证实、再按项目面向判断**,不被第三方博客的互相矛盾说法带偏;Claude 专属机制 vs 跨 agent 普适规范要分清归属。

### 更高杠杆的提问框架(本次交付给用户的方法论)

盲点扫描发现:问"设计/实现是否最佳"会诱导泛泛而答。更榨得出价值的问法——**按可逆性分类**(不可逆的死磕、可逆的快速试错)、**逼暴露隐藏假设**("依赖哪些未验证前提?先在哪崩?")、**要证伪路径而非评价**("给一条最便宜的实验证明它是错的")、**限定锚点**(窄问题挖深)、**设计与实现分开问**(前者问假设/可逆性,后者问边界/失败路径)。

## 复盘:Python → Rust Runtime + TS UI 语言/架构迁移(2026-07-06)

### 经过

一次"把项目从 Python 换语言"的任务,中途经历**两次方向掉头**:
1. Python → 纯 TypeScript(评估后:IO-bound、生态、部署形态 → 选 TS);
2. 纯 TS → **Rust Runtime + TS UI**(补充约束浮现:长运行 Runtime、本地+服务器双端、系统级隔离 → Runtime 定 Rust,UI 留 TS)。

最终落地:Cargo workspace 六层 crate + foundation/memory 契约、全文档 Rust 化、xtask 文档检查、覆盖率 93.43%,全链验证通过。

### 哪里费了 token / 返工(诚实记录)

| 返工点 | 浪费 | 根因 |
|---|---|---|
| 两次语言掉头 | 纯 TS 后端代码 + 14 篇文档的 TS 化**全部推倒重来**(翻两遍) | 语言决策前未先问清地基约束(部署形态/运行时寿命/系统级需求),第一次评估只覆盖了一部分维度 |
| 文档先 TS 化再 Rust 化 | 同批文档翻译两遍 | 方向未锁死就铺开大范围转换 |
| 子代理中途余额耗尽 | 2 个 agent 死掉、留半成品,需检测重派 | 未"先验证一个成功再批量扇出"(第二批才纠正) |
| cargo-llvm-cov 本地装不上 | 轮询等待十几分钟 | 重型/不确定安装未及早 time-box 并交用户 |
| model_gateway 命名 + 文档文件名 | 改名又改回(snake↔kebab 两轮) | 未先把"各类产物命名规范"一次定清(Rust 源码 snake vs 文档文件 kebab) |

### 做对的(保留)

- **大掉头前用结构化提问让用户拍板**——这是把返工成本控制住的关键,否则更贵。
- **第二批子代理"先验证一个成功再扇出"**——避免了整批浪费。
- **本地验证链(fmt/clippy/test)早跑**——问题当场暴露。
- **ADR 让设计资产跨语言零损失**——六层架构、事件协议、多租户不变量、memory 四件套全程未丢。

### 可迁移经验(→ 已提炼进根 AGENTS「协作效率原则」)

1. **地基决策先问全再动手**:架构/语言/存储/部署这类难逆转的选择,动手前一次性问清「部署形态·目标用户·运行时寿命·系统级需求」四类约束——一次问全会直接到终点,避免中途掉头推翻存量。
2. **方向未锁死前,不铺开可逆但昂贵的大范围转换**(全量改语言、全量改文档):等拍板再做,别做"很可能白费"的铺开。
3. **并行子代理:先确认一个成功,再批量扇出**;重型工具安装 time-box,失败早交用户,别死等。
4. **约定先立后用**:命名/风格规范一次定清再动手(Rust 源码 snake、文档文件 kebab、TS camelCase),避免改名往返。
5. **定向读取优于反复全量扫描**;复盘沉淀进本文件,不堆进每次载入的 AGENTS.md。

### 关于文档/记忆文件组织(本次采纳)

参照 Claude Code 官方 memory 最佳实践([support.claude.com](https://support.claude.com/en/articles/14553240-give-claude-context-claude-md-and-better-prompts)):根 `AGENTS.md` 每次会话全量载入、每行永久占 token,故只留**全局常用**;分场景细则放**子目录 `AGENTS.md`**(`crates/AGENTS.md`、`docs/AGENTS.md`),仅在该目录被触及时按需载入;完整复盘/参考材料放本文件,按需读取。单一事实源:命名等硬规则以 ADR 0014 为准,其余引用不重复维护。
