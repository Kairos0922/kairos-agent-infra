# 项目概述与设计目标

## 项目定位

**Kairos Agent Infra** 是一套为上层 Agent / 产品提供基础能力的**解耦基础设施**。它不是 Agent 框架,不规定 Agent 如何编排、如何调用 LLM;它只把 Agent 运行所需的**横切能力**(记忆、上下文,以及未来的其他模块)沉淀为彼此独立、可单独演进、可单独替换的模块。

一句话边界:**Agent 决定"做什么",Kairos 提供"记住什么、检索什么"。**

整个项目未来会包含多个 infra 模块(记忆、上下文等)。当前(第一阶段)只实现**记忆模块**,底座为后续模块提供共享的工程地基。

## 核心设计原则

### 1. 高内聚、低耦合

- **模块高内聚**:一个 infra 模块的领域逻辑、它依赖的存储/模型抽象、它的对外接口,都收在模块自己的目录里,自成一域、自包含。
- **模块间低耦合**:模块之间不互相 import 内部实现。记忆模块不感知未来的上下文模块,反之亦然。

### 2. 模块解耦、可插拔

- **模块与实现解耦**:模块依赖的向量库、embedding/rerank 模型都通过抽象接口接入。换实现不应改动模块的领域逻辑。
- **上层与 infra 解耦**:上层应用通过**适配层**调用 infra,不直接依赖 infra 内部结构。

### 3. 避免过度设计(YAGNI)

- **不替未来过度抽象**。只有当前阶段(底座 + 记忆模块)真正需要的东西才设计。
- **共享能力按需上提**:某个抽象(如 embedding 接口)目前只有记忆模块用,就放在记忆模块内;等出现第二个模块确有复用需求时,再评估上提到底座——而不是现在就猜一个全局接口。
- 底座只放**现在就真正横切**的东西(配置、错误、日志、trace、接口风格约定、工程化),不放业务逻辑,保持"薄"。

> **为什么这三条放在一起?** 它们互相制衡。"可插拔"容易滑向"为一切都做抽象层";"避免过度设计"把它拉回来——只为已经存在的复用需求做抽象。结果是:底座薄、模块自包含、抽象只在被证明需要时才出现。对一个要长期维护、要陆续加模块的"项目资产"来说,这样演进成本最低。

## 设计目标(本阶段)

| 目标 | 含义 | 验收信号 |
|------|------|---------|
| **模块自包含** | 记忆模块的领域逻辑、存储/模型抽象、API 都在模块内,不外泄到底座或其他模块。 | 删掉记忆模块目录,底座与项目骨架仍可独立存在。 |
| **实现可插拔** | 换向量库、换 embedding/rerank 模型,只改模块内的实现注入,不改领域逻辑。 | 记忆模块的向量库、embedding、rerank 都能通过配置切换。 |
| **三类记忆可用** | 语义、情景、程序,三类都能写入、检索、按各自策略淘汰。 | 三类各有独立的数据模型、写入/检索/淘汰路径。 |
| **统一检索层** | 向量、BM25、混合(RRF)、rerank 收敛到一套接口,对调用方屏蔽融合复杂度。 | 调用方只选"检索方法",不关心融合细节。 |
| **接口风格统一** | 同步/异步、错误处理、可观测性接入点在底座层面统一规定。 | 有成文约定,所有对外 API 一致。 |
| **底座支撑新模块** | 目录、配置、测试、日志/trace 接入点就位,新模块按骨架就能开工。 | 新模块无需重新设计工程结构。 |

## 非目标(Non-goals)

明确划出本阶段**不做**的事。这些不是"永远不做",而是"本阶段不做"——且按 YAGNI,不为它们预先设计接口。

| 非目标 | 说明 |
|--------|------|
| **上下文模块(context)** | 不实现。第二个模块出现时,作为验证底座可插拔性的真实案例(见 [roadmap](./roadmap.md))。 |
| **全局共享抽象层** | 不预先把 embedding/向量库等抽象提升为全局契约;它们留在记忆模块内,按需再上提。 |
| **工具/技能治理** | 不做。 |
| **多租户与权限体系** | 数据模型预留 `owner_id`/`namespace` 字段,但不实现完整隔离/鉴权/配额。 |
| **分布式 / 高并发** | 本阶段单机嵌入式形态(LanceDB 本地文件),不做多节点/水平扩展。 |
| **自动记忆演进/反思** | 不做后台自动合并、冲突消解、遗忘曲线。程序记忆提炼只做最小可用版本(规则门控 + 单次 LLM 抽取)。 |
| **图记忆 / 知识图谱** | 不做。 |
| **Web UI / 可视化** | 不做。可观测性以结构化日志 + trace 接入点为主。 |

## 与参考项目 EverOS 的关系(摘要)

我们实际克隆并分析了 [EverOS](https://github.com/EverMind-AI/EverOS)(`local-first memory runtime for agents`,commit `b7d15f7`)。完整分析在记忆模块文档 [everos-analysis](../modules/memory/everos-analysis.md),这里只给摘要:

**借鉴的**:派生索引可重建的思路、Provider Protocol + Factory 可插拔模型接入、方法→管线路由(对外少数检索方法、内部融合隐藏)、rerank 契约"只排序不过滤"。

**不同取舍**:不采用 Markdown 事实源(以 LanceDB 单一存储,降低双写+同步复杂度);算法不外置为私有二进制包(放仓库内、可读可改);本阶段不做 EverOS 式的四层融合管线(先用标准 RRF)。

> 这些借鉴点多与**记忆/检索的具体设计**相关,因此细节归在记忆模块文档里,而非项目层。这也是"模块的归模块"原则的体现。
